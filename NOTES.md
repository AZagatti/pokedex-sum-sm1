# Build Notes

- 2026-07-10: Starting build. Read SPEC.md. Empty dir, no git repo yet. Node v24.17.0, npm 11.17.0 available.
- 2026-07-10: `sv create` needed explicit sub-options (`vitest="usages:unit"`, `tailwindcss="plugins:none"`, `sveltekit-adapter="adapter:static"`) to skip all interactive prompts non-interactively — passing bare addon names still prompted for their sub-choices.
- 2026-07-10: SvelteKit 2.63 / vite-plugin-svelte 7.x resolve config differently than expected: passing `adapter`/`paths` directly to the `sveltekit()` Vite plugin call (available since kit 2.62) causes `svelte.config.js` to be ignored entirely. Deleted `svelte.config.js` and put all config in `vite.config.ts` to avoid two competing config sources silently fighting each other.
- 2026-07-10: `ultracite init --linter oxlint --pm npm` (via `CI=true` to force non-interactive) worked cleanly and wired oxlint+oxfmt with zero prompts — much smoother than expected from the docs.
- 2026-07-10: Oxlint only lints the `<script>` block of `.svelte` files, and doesn't understand Svelte 5 rune reactivity — `ultracite fix`'s `prefer-const` autofix repeatedly converted `let x = $state(...)` to `const x = $state(...)` for variables that ARE reassigned from template event handlers (mobileOpen, spriteVariant, errored, filtersOpen, sentinel). This is silently invalid Svelte and only surfaces as a `svelte-check` compile error ("Cannot assign to constant"), not an oxlint error. Had to disable `eslint/prefer-const` (and `func-style`, `sort-keys`, `filename-case`, a few Promise/loop style rules) project-wide in `oxlint.config.ts` since they fight idiomatic Svelte 5 + async worker-pool code. Lesson: after any `ultracite fix` pass, always rerun `npm run check` before trusting the formatter didn't break state mutation.
- 2026-07-10: Live-tested the Pokédex list page's "sort by base-stat total" feature (fetches full detail for all currently-filtered Pokémon) and it hung indefinitely at 649/1351 with no visible error. Root cause: PokeAPI returns `cries.legacy: null` for a subset of Pokémon, but the Zod schema required `z.string()` — a thrown ZodError inside an unawaited/uncaught path silently killed the concurrent fetch batch. Fixed by making `cries.latest`/`legacy` nullable in the schema, wrapping the per-item fetch in try/catch (so one bad entry doesn't stall the whole sort), and adding a live X/Y progress indicator so future stalls are visible immediately instead of looking identical to "still working."
- 2026-07-10: A general-purpose subagent code review caught a real race condition in the sort-by-stats and filter-set-loading effects (rapid user interaction could let a stale async computation overwrite state computed against a newer filter selection) — added monotonic request tokens to both. Also fixed: dead ternary in favorites page pluralization, unguarded `localStorage` calls in store mutators (only the initializers checked `typeof localStorage === 'undefined'`), missing `role="img"`/`aria-label` on the broken-image fallback, and inconsistent use of raw `<img>` vs the app's `<PokemonImage>` fallback-handling component on the berries pages.
- 2026-07-10: Playwright e2e tests initially failed 3-4 of 10 with confusing errors ("element(s) not found" with no other detail) when navigating directly to `/favorites`, `/berries`, or a 404 route under `vite preview`. Root cause was NOT a routing/SPA-fallback bug (`vite preview` serves the adapter-static output correctly) — it was `baseURL` in `playwright.config.ts` missing a trailing slash (`http://localhost:4173/pokedex-sum-sm1` instead of `.../pokedex-sum-sm1/`), so `page.goto("/berries")` (leading slash) went to `localhost:4173/berries` and `page.goto("./berries")` under a base without a trailing slash resolved to `localhost:4173/berries` too (standard URL-resolution behavior, same as `new URL('./x', 'http://h/base')` dropping the last segment). Playwright's own error-context snapshot actually named the fix ("did you mean to visit /pokedex-sum-sm1/berries instead?") — should have read that message first instead of assuming an app bug.
- 2026-07-10: `curl` against `localhost` from the Bash tool intermittently timed out (exit 28) even though the dev/preview server logs showed it was up and the browser tooling could reach it fine — apparent sandbox networking quirk specific to this environment, not a real server issue. Stopped trusting curl as a liveness check partway through and relied on the browser tools instead.
